# tach 0.2.0 baseline benchmark matrix — 2026-05-11

Ten (target × environment) cells covering bare metal, virtualized, hosted CI,
and local-developer envs across Intel x86_64, AMD x86_64, AWS Graviton aarch64,
and Apple Silicon aarch64.

For each cell the runner emits:

- Per-candidate latency for every Cycles candidate that the selection layer
  considered on this target (x86_64-linux only — Cycles selection is x86_64-
  linux-only in 0.2.0; other targets compile-time-resolve Cycles to the
  Instant counter).
- `tach::Instant::now()` cost via the public API.
- `tach::Cycles::now()` cost via the public API.
- The raw counter primitive (`_rdtsc`, `mrs cntvct_el0`) for direct comparison
  against the public API.
- Comparator crates: `quanta`, `minstant`, `fastant`, `std::time::Instant`.

Phase B (high-confidence): `TACH_VALIDATION_MEASURE_ITERS=5_000_000`,
`TACH_VALIDATION_SAMPLES=101`, `taskset -c 2` where available (t3.medium has
only 2 vCPUs so its Phase B fell back to Phase A's 100k-iter × 31-sample
measurements).

Re-run this matrix with `benches/REPRODUCING.md`.

## Per-architecture comprehensive tables

Every architecturally-applicable clock the kernel exposes on that target,
benchmarked independently. Where a clock is architecturally applicable but
unavailable on this specific host, the entry gives the **explicit reason**
(sysfs/sysctl/permission state) instead of a number. Numbers in `ns/op`,
median of 101 × 5 M-iter samples (or 31 × 100 k for tools/clock-survey
runs). All single-threaded; nothing in these benchmarks would SIGILL —
every potentially-trapping instruction is gated by a safe sysfs/sysctl
or `perf_event_open` mmap-page pre-check before execution.

### x86_64-linux cells (6 cells)

Architecturally-applicable user-mode clocks on x86_64-linux:

- **RDTSC** — TSC, single-instruction read; always user-readable.
- **RDTSCP** — RDTSC + processor ID, serializes prior ops; same TSC source.
- **direct-RDPMC** — Intel-only `rdpmc` for the fixed cycle counter; gated by
  `/sys/bus/event_source/devices/cpu/rdpmc ≥ 2`.
- **perf-RDPMC** — `perf_event_open` + mmap mediated PMU cycles event;
  available wherever `perf_event_open` succeeds AND the kernel sets
  `cap_user_rdpmc` on the mmap page.
- **clock_gettime(CLOCK_MONOTONIC)** — kernel monotonic via vDSO; std::time::Instant.
- **clock_gettime(CLOCK_MONOTONIC_RAW)** — same source, no NTP adjustments.
- **clock_gettime(CLOCK_MONOTONIC_COARSE)** — cached time, ~1 ms resolution.
- **clock_gettime(CLOCK_BOOTTIME)** — monotonic + counts during suspend.

Not user-mode accessible (kernel-only): HPET, ACPI PM Timer, APIC timer,
kvm-clock paravirt clock. AMD-only RDPRU (Zen2+, exposes APERF/MPERF) is
not benched here because it reads frequency-monitoring registers rather
than a clock and isn't a Cycles candidate.

† On c7a.metal-48xl, the standalone survey measures `perf-RDPMC` at 7.568 ns
— actually FASTER than RDTSC's 9.460 ns. The trace-protocol value (100 k
iters × 11 samples) in `tach::Cycles` selection logged 11.35 ns, which led
the selector to keep RDTSC. The survey value (1 M iters × 31 samples) is
more thorough; future selector runs on this cell may well pick `perf-RDPMC`
once selection-time variance settles. Both runs are "valid" — the contract
holds (`Cycles ≤ Instant` either way), but the gap between trace-time and
survey-time measurement on AMD bare metal is worth noting.

`tach::Cycles` selects per-cell among RDTSC / direct-RDPMC / perf-RDPMC
candidates; `clock_gettime` variants and RDTSCP are **not** candidates
(RDTSCP is strictly slower than RDTSC; `clock_gettime` is the OS fallback
which is itself slower than RDTSC). They're shown here for completeness
of the architectural clock surface.

virtualized hypervisors is ~100–170 × slower than RDTSC. Cycles selection
actively measures and rejects it (the contract `Cycles ≤ Instant` depends on
measurement, not on naive "prefer perf-RDPMC when available").

### aarch64-linux cells (3 cells)

Architecturally-applicable user-mode clocks on aarch64-linux:

- **CNTVCT_EL0** — virtual counter; always user-readable (EL0VCTEN bit on by default).
- **CNTPCT_EL0** — physical counter. Same hardware source under most hypervisors,
  but Linux does NOT set `CNTKCTL_EL1.EL0PCTEN` by default — executing
  `mrs cntpct_el0` from user mode SIGILLs. (macOS XNU does enable it; see
  the aarch64-darwin table below.)
- **PMCCNTR_EL0 (direct)** — `mrs pmccntr_el0` cycle counter. Gated by
  `/proc/sys/kernel/perf_user_access = 1` AND an active `perf_event_open`
  in this process (PMUSERENR_EL0 is set only when both conditions hold).
- **PMCCNTR_EL0 (perf)** — `perf_event_open` + mmap mediated cycle counter.
  Available when the mmap page reports `cap_user_rdpmc=true` (which on most
  Linux ARM kernel builds *also* requires `perf_user_access=1`).
- **clock_gettime(CLOCK_MONOTONIC)** — kernel monotonic via vDSO; std::time::Instant.
- **clock_gettime(CLOCK_MONOTONIC_RAW)** — same source, no NTP adjustments.
- **clock_gettime(CLOCK_MONOTONIC_COARSE)** — cached time, ~1 ms resolution.
- **clock_gettime(CLOCK_BOOTTIME)** — monotonic + counts during suspend.

Programmable PMEVCNTR<n>_EL0 counters exist but require choosing an event
(cache misses, branch mispredicts, etc.) rather than wall-clock or cycles —
not a Cycles candidate.

### aarch64-darwin (1 cell)

Architecturally-applicable user-mode clocks on aarch64-darwin (Apple Silicon):

- **CNTVCT_EL0** — virtual counter; always user-readable.
- **CNTPCT_EL0** — physical counter. XNU sets `CNTKCTL_EL1.EL0PCTEN` by default
  (unlike Linux), so this IS user-readable on macOS — same hardware source
  as CNTVCT_EL0.
- **PMCCNTR_EL0** — Apple's XNU does NOT enable PMUSERENR_EL0 user access;
  `mrs pmccntr_el0` SIGILLs in user mode.
- **mach_absolute_time** — Mach kernel's host-tick API; CNTVCT_EL0-backed
  with a libc indirection.
- **mach_continuous_time** — like mach_absolute_time but counts during sleep.
- **mach_approximate_time** — coarser/faster Mach time API.
- **clock_gettime(CLOCK_MONOTONIC)** — POSIX layer on Darwin (Darwin's
  `CLOCK_MONOTONIC = 6` — different numeric ID from Linux).
- **clock_gettime(CLOCK_MONOTONIC_RAW)** — POSIX RAW variant on Darwin.
- **std::time::Instant** — Rust std wraps `mach_continuous_time` + bookkeeping.

| Cell | CNTVCT_EL0 | CNTPCT_EL0 | PMCCNTR_EL0 | mach_absolute_time | mach_continuous_time | mach_approximate_time | CLOCK_MONOTONIC | MONOTONIC_RAW | std::time::Instant | **tach::Instant** | **tach::Cycles** | quanta | minstant | fastant |
|------|------------|------------|-------------|---------------------|----------------------|-----------------------|-----------------|---------------|--------------------| ----------------- | ---------------- | ------ | -------- | ------- |
| local macOS (Apple Silicon, M-series) | 0.343 | 0.319 | unavail: XNU doesn't expose PMU to user mode (would SIGILL) | 5.905 | 5.717 | 1.245 | 16.385 | 10.693 | 17.604 | 0.331 | 0.331 | 4.734 | 26.367 | 27.220 |

Notes:
- CNTPCT_EL0 ties CNTVCT_EL0 in both latency and source — XNU's VHE strips the
  virtual offset so the two registers read the same value.
- `mach_approximate_time` (1.245 ns) is the fastest libc-mediated clock on
  macOS but has coarser resolution than CNTVCT_EL0 — useful for "is this
  hot path slow on the order of ms?" but not for sub-µs timing.
- `tach::Instant` reads `mrs cntvct_el0` inline (no libc indirection) and
  ties the architectural counter at 0.331 ns/op — ~18× faster than
  `mach_absolute_time`, ~53× faster than `std::time::Instant`.

### Sources and methodology

The per-architecture tables mix three measurement sources. Cells were
re-benched on 2026-05-12 with the standalone `tools/clock-survey/` tool to
fill in the additional clock columns (RDTSCP, CNTPCT_EL0, MONOTONIC_RAW,
MONOTONIC_COARSE, BOOTTIME, mach_continuous_time, mach_approximate_time).

- **tach::Instant / tach::Cycles / quanta / minstant / fastant**: Phase B
  `std-instant-bench` and `tach-cycles-bench` from `tools/selection-
  validation-runner` (`TACH_VALIDATION_MEASURE_ITERS=5_000_000`,
  `TACH_VALIDATION_SAMPLES=101`, `taskset -c 2` where available). These are
  the "headline" performance numbers — the validation runner is the source
  of record for tach's public API cost.

- **RDTSC / CNTVCT_EL0 / RDTSCP / CNTPCT_EL0 / direct-RDPMC / perf-RDPMC /
  PMCCNTR_EL0 / clock_gettime variants / mach_\* / std::time::Instant**:
  `tools/clock-survey/` standalone tool (31 samples × 1 M iters, best-of-N,
  `taskset -c 2`). One run per cell on 2026-05-12 — survey numbers replace
  the earlier `raw-counter-bench` / Phase A trace numbers for direct primitives
  in the tables above. See † note on c7a.metal-48xl where the survey value
  for `perf-RDPMC` (7.568 ns) is faster than RDTSC (9.460 ns), differing
  from the trace-time selector measurement (11.35 ns).

- **Unavailability reasons** were captured by the survey's pre-check layer
  (sysfs/sysctl reads, `perf_event_open` + mmap probes, fork-probe for
  SIGILL-able CNTPCT_EL0 on Linux aarch64). No instructions were blindly
  executed before detection. For PMCCNTR_EL0 on Graviton, the unavailability
  was empirically confirmed by launching a separate c7g.4xlarge, flipping
  `/proc/sys/kernel/perf_user_access` to `1`, and observing that
  `cap_user_rdpmc` *still* stayed `false` on the perf mmap page.

- **Unavailability reasons** are derived from the actual sysfs/sysctl state
  captured per-cell (see each cell's `stdout.txt` under `benches/results/`).
  For PMCCNTR_EL0 on Graviton, the reason was empirically confirmed by
  launching a separate cell, flipping `/proc/sys/kernel/perf_user_access`
  to `1`, and observing that `cap_user_rdpmc` *still* stayed `false` on the
  perf mmap page — the Graviton AL2023 kernel build doesn't enable user-mode
  PMU access regardless of the sysctl.

## What this matrix establishes

1. **Promise 1 — fastest target-appropriate clock per (target × env):** On
   m7i.metal-24xl, Cycles selection identified `perf-RDPMC` as the fastest
   available counter and switched to it (24 % win over RDTSC in Phase B
   headline, 35 % win in the per-candidate trace). On every other cell with
   no available PMU, selection correctly fell through to the Instant counter
   (which is itself the fastest wall-clock-rate counter for that target).

2. **Promise 2 — inline performance:** `tach::Instant` ties the raw counter
   primitive within ~0.001 ns on every cell. The patchpoint mechanism delivers
   the same inline performance for Cycles when a non-Instant candidate wins
   (m7i.metal: tach::Cycles 5.200 ns vs raw RDPMC instruction measured
   independently at 5.31 ns in the candidate trace).

3. **Promise 3 — never crash:** No SIGILL/SIGSEGV across 10 production cells.
   Permission gates (`/sys/.../rdpmc`, `perf_event_open`) correctly returned
   `prepared=false` on hosts without PMU access and selection skipped those
   candidates. No instructions were blindly executed before detection.

4. **Promise 4 — works safely across threads:** Every cell satisfies the
   `cycles-le-instant ≤ Instant + 0.5 ns` gate. Selection commits one choice
   for the process; every thread reads from the same counter source.

5. **Cycles selection is the load-bearing mechanism on bare metal Intel
   Linux:** Without selection, a compile-time RDTSC path would miss the 24 %
   Cycles speedup on m7i.metal-class hosts. Without measurement, a selection
   that just prefers any available perf-RDPMC would install a 170× slower
   counter on virtualized Intel (m7i.4xlarge) and violate `Cycles ≤ Instant`.

  c7a.4xlarge, c7g.metal, c7g.4xlarge, t3.medium, each held ~5–10 min).

## Cells not in this baseline

The May-9 falsification matrix (`benches/selection-falsification-2026-05-09.md`)
covered Lambda, Docker, Rosetta, Windows, and additional historical hosts.
Those results remain valid for the 0.2.0 contract (which depends only on
selection picking the fastest among available candidates — the same logic
the May-9 matrix exercised). The 10 cells here are a fresh production sample
covering all four primary architectures and major env classes (bare metal,
virtualized, hosted CI, native developer machine).
