# tach 0.2.0 baseline benchmark matrix — 2026-05-11

This baseline run verifies the contract for `tach 0.2.0` across a representative
production sample: `Instant::now()` picks the fastest wall-clock-rate counter
available on each target, `Cycles::now()` picks the fastest counter overall
(falling back to the Instant counter when no PMU is faster), and on every cell
**`tach-cycles-bench ≤ tach-instant-bench + 0.5 ns`** (the `cycles-le-instant`
contract — the gate the reproducible matrix infrastructure checks).

Re-run this matrix with `benches/REPRODUCING.md`. The orchestration scripts
under `benches/scripts/` are checked-in and idempotent.

Values are median ns/op from Phase B (5 M iterations × 101 samples, `taskset -c 2`
where available). `cycles-le-instant` is the contract gate.

| Cell | Target | Env | Instant clock | Cycles clock | Instant | Cycles | Δ | cycles-le-instant |
|---|---|---|---|---|---|---|---|---|
| AWS m7i.metal-24xl | x86_64-linux-gnu | Intel bare metal | x86_64-rdtsc | **x86_64-perf-rdpmc** | 6.843 | 5.200 | **−24 %** | **pass** |
| GitHub ubuntu-24.04 | x86_64-linux-gnu | AMD VM | x86_64-rdtsc | x86_64-rdtsc | 8.927 | 8.928 | 0 % | pass |
| GitHub ubuntu-24.04-arm | aarch64-linux-gnu | aarch64 VM | aarch64-cntvct | aarch64-cntvct | 13.356 | 13.356 | 0 % | pass |
| Local macOS | aarch64-darwin | Apple Silicon | aarch64-cntvct | aarch64-cntvct | 0.331 | 0.331 | 0 % | pass |

**Every cell satisfies the contract.** On `m7i.metal-24xl`, Cycles selection
correctly picks `perf-RDPMC` over RDTSC (the only env in this baseline with
PMU access exposed), delivering ~24 % lower read cost. On the other cells,
Cycles falls back to the wall-clock-rate counter Instant uses, satisfying the
`Cycles ≤ Instant` guarantee trivially.

## Per-cell trace

The per-candidate selection trace is captured in the corresponding `phase-a.log`
under `benches/results/<cell>/`. Salient observations:

### AWS m7i.metal-24xl (x86_64-linux Intel bare metal)

- `/sys/bus/event_source/devices/cpu/rdpmc = 2` (kernel grants unprivileged RDPMC).
- direct-RDPMC: prepared, validated, 5.245 ns/op.
- perf-RDPMC: prepared, validated, 5.200 ns/op (winner).
- RDTSC: prepared, validated, 6.842 ns/op (would have been winner without PMU).

Selection picks `x86_64-perf-rdpmc` and the patchpoint mechanism rewrites every
`Cycles::now()` callsite to inline the perf-event-mediated PMU read. After
first call, the hot path costs the raw counter read with no dispatch overhead.

### GitHub ubuntu-24.04 (x86_64-linux AuthenticAMD virtualized)

- `/sys/bus/event_source/devices/cpu/rdpmc` not present.
- direct-RDPMC, perf-RDPMC: `prepared=false`.
- RDTSC: prepared, validated, 8.928 ns/op (winner).

Cycles falls back to RDTSC = Instant. `cycles-le-instant` holds trivially.

### GitHub ubuntu-24.04-arm (aarch64-linux VM)

- `/proc/sys/kernel/perf_user_access = 0`.
- Cycles selection is not compiled on aarch64-linux in 0.2.0 (see PROJECT.md
  "Open work"). `Cycles::now()` compile-time-resolves to `arch::ticks()`
  which reads CNTVCT_EL0.

### Local macOS (aarch64-darwin Apple Silicon)

- No user-mode PMU on Apple Silicon.
- Both `Instant::now()` and `Cycles::now()` compile-time-resolve to `mrs cntvct_el0`.
- 0.331 ns/op on M-series CPU.

## What this baseline establishes for 0.2.0

1. **`Instant`** compile-time direct dispatch hits the raw counter cost on every
   measured target. Inline performance honored.
2. **`Cycles`** selection on Linux x86_64 correctly identifies and switches to
   `perf-RDPMC` when PMU permission is granted (m7i.metal), and correctly falls
   through to RDTSC when not (GitHub AMD VM). Contract `Cycles ≤ Instant` holds.
3. **Cycles on aarch64-linux / riscv64-linux** falls back to the Instant counter
   (compile-time = Instant). This is the documented 0.2.0 scope — no PMU-based
   patching for these architectures yet. The contract holds because Cycles
   *is* Instant on these targets.
4. **No thread spawning at startup** anywhere in the library hot path. The
   selection on x86_64-linux is single-threaded; the cross-thread monotonicity
   property is dropped from the Instant contract (use `std::time::Instant` if
   you need it).

## Cells not in this baseline

The full plan listed 13 cells; this baseline covers 4 to validate the contract
without burning the full AWS budget. Cells deferred (not blocking 0.2.0):

- AWS m7i.4xlarge — virtualized Intel; expected to match the GitHub AMD VM
  pattern (Cycles = RDTSC fallback). Documented as similar in the May-9
  falsification matrix.
- AWS c7a.metal-48xl, c7a.4xlarge — AMD Zen4 bare metal / virtualized. Cycles
  falls back to RDTSC on both (May-9 confirmed).
- AWS c7g.metal, c7g.4xlarge — Graviton bare metal / virtualized. Cycles =
  Instant on aarch64-linux in 0.2.0 regardless.
- AWS Lambda x86_64 / aarch64 — Firecracker; Cycles falls back per May-9.
- Docker riscv64 / loongarch64 — emulated; Cycles = Instant in 0.2.0.

Re-running these cells via `benches/scripts/run-matrix.sh` after adding rows
to `AWS_CELLS` will produce extended baselines without touching code.
