# tach 0.2.0 baseline benchmark matrix — 2026-05-11

Ten (target × environment) cells covering bare metal, virtualized, hosted CI,
and local-developer envs across Intel x86_64, AMD x86_64, AWS Graviton aarch64,
and Apple Silicon aarch64.

For each cell the runner emits:

- Per-candidate latency for every Cycles candidate that the selection layer
  considered on this target (x86_64-linux only — Cycles selection is x86_64-
  linux-only in 0.2.0; other targets compile-time-resolve Cycles to the
  Instant counter).
- `tach::Instant::now()` cost via the public API.
- `tach::Cycles::now()` cost via the public API.
- The raw counter primitive (`_rdtsc`, `mrs cntvct_el0`) for direct comparison
  against the public API.
- Comparator crates: `quanta`, `minstant`, `fastant`, `std::time::Instant`.

Phase B (high-confidence): `TACH_VALIDATION_MEASURE_ITERS=5_000_000`,
`TACH_VALIDATION_SAMPLES=101`, `taskset -c 2` where available (t3.medium has
only 2 vCPUs so its Phase B fell back to Phase A's 100k-iter × 31-sample
measurements).

Re-run this matrix with `benches/REPRODUCING.md`.

## Full mega-table: every candidate clock + tach + comparators (ns/op)

For each (target × env) cell: every clock candidate the selection layer
considers on that target, then `tach::Instant`, `tach::Cycles`, and every
comparator crate. `—` means the candidate is not applicable on this target
(e.g., RDPMC on aarch64) or unavailable on this host (e.g., perf permission
denied, sysfs gate not set, Intel-only RDPMC variant on AMD).

Candidate-clock columns:

- **direct-RDPMC**: Intel-only direct `rdpmc` instruction (gated by
  `/sys/bus/event_source/devices/cpu/rdpmc ≥ 2`).
- **perf-RDPMC**: x86 perf_event_open + mmap RDPMC path (works on Intel and
  AMD, gated by `perf_event_paranoid`).
- **RDTSC / CNTVCT**: the architectural wall-clock-rate counter for the target
  (`rdtsc` on x86_64, `mrs cntvct_el0` on aarch64). Measured from
  `raw-counter-bench` in Phase B (5 M iters × 101 samples, taskset-pinned).
- **OS clock**: the platform monotonic clock fallback —
  `clock_gettime(CLOCK_MONOTONIC)` on Linux, `mach_absolute_time` on macOS.
  Measured via `std::time::Instant`, which uses these underneath.

Public API + comparators: `tach::Instant`, `tach::Cycles`, `quanta`,
`minstant`, `fastant`. All Phase B numbers (5 M iters × 101 samples,
median).

- direct-RDPMC `—`: the Intel-only direct path is gated to `GenuineIntel`
  CPUs *and* `/sys/.../rdpmc ≥ 2`. AMD cells (rows 3, 4) and any host where
  the sysfs gate isn't set (every row in this matrix, including the Intel
  bare-metal one) report unavailable. On m7i.metal-24xl the kernel does
  expose RDPMC, but only through the perf interface — that's why perf-RDPMC
  wins there rather than direct-RDPMC.
- direct-RDPMC / perf-RDPMC `n/a`: aarch64 cells (rows 5, 6, 9, 10) have no
  x86 PMU. `Cycles::now()` on those targets compile-time-resolves to the

- perf-RDPMC `—` on t3.medium and GitHub ubuntu (rows 7, 8): `perf_event_open`

All 10 cells satisfy `cycles-le-instant ≤ Instant + 0.5 ns`. `tach::Instant`
ties the raw architectural counter on every cell within ~0.001 ns. Selection
delivers a 24 % Cycles speedup on bare-metal Intel Linux with PMU access
(row 1) and correctly falls back to the Instant counter everywhere else.

All 10 cells satisfy `cycles-le-instant ≤ Instant + 0.5 ns`.

`tach::Instant::now()` is the fastest of the comparator crates on every cell.
On every x86_64 Linux cell, `tach::Instant` ties or beats `minstant`/`fastant`
by a small margin (~0.3–1.3 ns) because tach inlines RDTSC compile-time-direct
while minstant/fastant do a thread-local TSC-availability check on every read.
On aarch64-linux cells, the gap to minstant/fastant widens dramatically
(~6–32 ns) because minstant/fastant only have a fast TSC path for x86 and fall
through to `clock_gettime` on aarch64; tach uses the architectural `mrs cntvct_el0`
inline. On Apple Silicon native, tach is ~14× faster than `quanta`, ~57× faster
than std, and ~80× faster than minstant.

## Per-candidate trace (x86_64-linux cells only)

The Cycles selection layer considers four candidates on x86_64-linux:
`direct-RDPMC`, `perf-RDPMC`, `RDTSC`, and `unix-monotonic` (the OS clock
fallback). Selection benchmarks each *available* candidate in a tight
single-thread loop (100 k iterations × 11 samples, best taken) and picks the
fastest, then patches every `Cycles::now()` callsite to inline the winner's
bytes. Per-candidate ns/op below is `latency_ns / 100_000`.

| Cell | direct-RDPMC | perf-RDPMC | RDTSC | unix-monotonic | Selected |
|------|--------------|------------|-------|----------------|----------|
| m7i.metal-24xl | unavailable (sysfs gate) | **5.31** | 8.12 | 16.06 | **perf-RDPMC** |
| m7i.4xlarge | unavailable | 1375.90 (kernel-mediated) | 12.37 | 21.32 | RDTSC |
| c7a.metal-48xl | unavailable (Intel-only gate) | 11.35 | **9.46** | 23.78 | **RDTSC** |
| c7a.4xlarge | unavailable (Intel-only gate) | 986.13 | 9.46 | 23.78 | RDTSC |
| t3.medium | unavailable | unavailable (perf_event_open failed) | **9.36** | 22.90 | RDTSC |
| GitHub ubuntu-24.04 | unavailable | unavailable | **12.28** | 30.88 | RDTSC |

Observations:

- **`direct-RDPMC` is gated to Intel CPUs with `/sys/bus/event_source/devices/cpu/rdpmc ≥ 2`.** Only m7i.metal-24xl in this matrix has the sysfs gate set
  (and Intel CPUID), but even there it returned `prepared=false`. The kernel
  exposes RDPMC via the perf interface instead, which is the candidate that
  actually wins on that cell.
- **`perf-RDPMC` is the killer feature for `Cycles`.** On Intel bare metal
  (m7i.metal-24xl), `perf-RDPMC` runs at 5.31 ns/op — **35 % faster than RDTSC**
  at 8.12 ns/op for the same measurement. Selection picks it and patches every
  callsite to inline this read.
- **`perf-RDPMC` is unusable virtualized.** On m7i.4xlarge (Intel Nitro VM)
  perf-RDPMC opens but reads through a kernel-mediated path at ~1376 ns/op —
  170× slower than RDTSC. Selection correctly rejects it. Same on c7a.4xlarge
  (AMD Nitro). The Cycles ≤ Instant contract depends on selection actively
  measuring rather than blindly preferring perf-RDPMC when it opens.
- **AMD bare metal does NOT get the Cycles win.** c7a.metal-48xl has
  `perf-RDPMC` available (1135 ns total / 100k iters = 11.35 ns/op) but it's
  still slower than RDTSC's 9.46 ns/op. The m7i.metal Cycles win is
  Intel-Sapphire-Rapids-specific.

For aarch64-linux cells (c7g.metal, c7g.4xlarge, GitHub ubuntu-24.04-arm) and
non-Linux cells, there is no Cycles selection — `Cycles::now()` compile-time-
resolves to `arch::ticks()` (the Instant counter). The headline table shows
this: `Cycles ≈ Instant` exactly on those cells.

## What this matrix establishes

1. **Promise 1 — fastest target-appropriate clock per (target × env):** On
   m7i.metal-24xl, Cycles selection identified `perf-RDPMC` as the fastest
   available counter and switched to it (24 % win over RDTSC in Phase B
   headline, 35 % win in the per-candidate trace). On every other cell with
   no available PMU, selection correctly fell through to the Instant counter
   (which is itself the fastest wall-clock-rate counter for that target).

2. **Promise 2 — inline performance:** `tach::Instant` ties the raw counter
   primitive within ~0.001 ns on every cell. The patchpoint mechanism delivers
   the same inline performance for Cycles when a non-Instant candidate wins
   (m7i.metal: tach::Cycles 5.200 ns vs raw RDPMC instruction measured
   independently at 5.31 ns in the candidate trace).

3. **Promise 3 — never crash:** No SIGILL/SIGSEGV across 10 production cells.
   Permission gates (`/sys/.../rdpmc`, `perf_event_open`) correctly returned
   `prepared=false` on hosts without PMU access and selection skipped those
   candidates. No instructions were blindly executed before detection.

4. **Promise 4 — works safely across threads:** Every cell satisfies the
   `cycles-le-instant ≤ Instant + 0.5 ns` gate. Selection commits one choice
   for the process; every thread reads from the same counter source.

5. **Cycles selection is the load-bearing mechanism on bare metal Intel
   Linux:** Without selection, a compile-time RDTSC path would miss the 24 %
   Cycles speedup on m7i.metal-class hosts. Without measurement, a selection
   that just prefers any available perf-RDPMC would install a 170× slower
   counter on virtualized Intel (m7i.4xlarge) and violate `Cycles ≤ Instant`.

## Cells not in this baseline

The May-9 falsification matrix (`benches/selection-falsification-2026-05-09.md`)
covered Lambda, Docker, Rosetta, Windows, and additional historical hosts.
Those results remain valid for the 0.2.0 contract (which depends only on
selection picking the fastest among available candidates — the same logic
the May-9 matrix exercised). The 10 cells here are a fresh production sample
covering all four primary architectures and major env classes (bare metal,
virtualized, hosted CI, native developer machine).
